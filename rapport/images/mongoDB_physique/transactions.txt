// ============================================================================
// RÈGLES DE TRANSITION D’ÉTAT: status vs refundStatus
// ============================================================================
//
// **Principe central :**
// - `status` suit le cycle de vie du PAIEMENT
// - `refundStatus` suit le cycle de vie du REMBOURSEMENT (indépendant mais lié)
//
// **Règles de transition :**
//
// 1. **Flux de paiement :**
//    PENDING → SUCCEEDED (le webhook confirme le paiement)
//    PENDING → FAILED (le webhook signale un échec ou timeout)
//    PENDING → CANCELLED (annulation confirmée par le provider ; l’Investment associé
//    peut être CANCELLING le temps de l’annulation, mais Transaction.status ne passe jamais à CANCELLING)
//
// 2. **Flux de remboursement (à partir de status=SUCCEEDED) :**
//    refundStatus : NOT_ATTEMPTED → PENDING (remboursement initié)
//    refundStatus : PENDING → SUCCEEDED (provider confirme le remboursement)
//    refundStatus : PENDING → FAILED (provider rejette le remboursement)
//    refundStatus : FAILED → PENDING (relance manuelle admin)
//
// 3. **Règles autoritaires :**
//    - status PEUT valoir "REFUNDED" UNIQUEMENT si refundStatus = "SUCCEEDED"
//    - Lorsque refundStatus → SUCCEEDED :
//        • status → "REFUNDED"
//        • refundedAt → horodatage courant
//        • Investment.status → "REFUNDED"
//    - Si refundStatus = "FAILED" :
//        • status conserve sa valeur précédente (SUCCEEDED ou FAILED)
//        • FailedRefundEvent créé pour traitement admin
//        • Investment.status reste inchangé jusqu’à résolution
//
// 4. **Plusieurs tentatives de remboursement :**
//    - refundAttempts incrémenté à chaque relance
//    - lastRefundAttemptAt mis à jour à chaque tentative
//    - L’admin peut relancer indéfiniment via UC_RetryRefund
//    - Chaque relance : refundStatus FAILED → PENDING, refundAttempts++
//
// 5. **Combinaisons interdites (vérifiées côté service) :**
//     status="REFUNDED" + refundStatus="NOT_ATTEMPTED" (impossible)
//     status="REFUNDED" + refundStatus="PENDING" (le remboursement doit être confirmé)
//     status="REFUNDED" + refundStatus="FAILED" (contradictoire)
//     status="SUCCEEDED" + refundStatus="SUCCEEDED" → bascule vers status="REFUNDED"
//
// 6. **refundStatus invalide pour paiements échoués :**
//    Si transaction.status = "FAILED", refundStatus DOIT rester "NOT_ATTEMPTED".
//    Aucun remboursement ne doit être initié sur un paiement échoué. Le service DOIT l’interdire.
//
// **Logique de relance :**
// - Relance paiement : créer une NOUVELLE Transaction sous le même Investment, attemptNumber++.
// - Relance remboursement : mettre à jour la MÊME Transaction, refundAttempts++.
//
// ============================================================================

// ============================================================================
// MODÈLE DE RELANCE (clarification)
// ============================================================================
// - Un Investment peut avoir plusieurs Transactions (relances).
// - Investment.paymentAttempts trace le total des tentatives.
// - Chaque Transaction.attemptNumber correspond à une tentative.
// - Quand un paiement réussit, Investment.status devient SUCCESS,
//   et aucune nouvelle Transaction n’est créée pour cet Investment.
// - Un paiement échoué laisse l’Investment en FAILED; l’utilisateur peut relancer,
//   ce qui crée une nouvelle Transaction sous le même Investment.
// ============================================================================
