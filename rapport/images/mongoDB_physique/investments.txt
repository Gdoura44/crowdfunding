// RÈGLES DE VALIDATION (appliquées au niveau service / Mongoose pre-save)
// ============================================================================
//
// **Règle 1: cancellingStartedAt doit être défini quand status = CANCELLING**
//   - Au passage en 'CANCELLING', cancellingStartedAt DOIT être défini à l’instant courant.
//   - En sortie de 'CANCELLING', cancellingStartedAt DOIT être effacé (null).
//
// cancellationGracePeriodMinutes: lu par le cron expire-projects pour déterminer
// quand un investissement SUCCESS ne peut plus être annulé.
// La valeur est par investissement (pas au niveau projet) pour rester flexible.
//
// Implémentation (hook Mongoose pre-save):
//
//   InvestmentSchema.pre('save', function(next) {
//     if (this.status === 'CANCELLING' && !this.cancellingStartedAt) {
//       return next(new Error('cancellingStartedAt requis quand status = CANCELLING'));
//     }
//     if (this.status !== 'CANCELLING' && this.cancellingStartedAt) {
//       this.cancellingStartedAt = null;
//     }
//     next();
//   });
//
// **Règle 2: nettoyage scheduledForDeactivation**
//   - Le flag scheduledForDeactivation DOIT être effacé après succès/échec de la désactivation.
//   - Le cron (cleanup-stuck-states) efface les flags “stale” de plus de 5 minutes.
//   - Ce flag est mis à true pendant la désactivation admin d’un projet
//    (see code_admin_deactivate_project.txt)
//     et remis à false par le cron de réconciliation (cleanup-stuck-states) après traitement ou timeout.
//
// **Règle 3: relance de FAILED vers INITIATED**
// - Si Investment.status = FAILED, il peut repasser à INITIATED uniquement dans le cadre d’une relance utilisateur.
// - Cette transition DOIT s’accompagner de la création d’une nouvelle Transaction
//   et de l’incrément de paymentAttempts dans la même opération atomique.
// - Le cron d’expiration se base sur createdAt de la DERNIÈRE transaction,
//   pas sur Investment.createdAt, pour ne pas expirer à tort une relance.
//   **Effet d’une annulation quand project.status = FUNDED :**
// - L’annulation est autorisée seulement si l’investissement est encore dans sa période de grâce.
// - Après annulation, le système doit:
//    1. Marquer investment = CANCELLED, transaction = CANCELLED.
//    2. Recalculer le total financé (somme des montants des investissements SUCCESS).
//    3. Si le nouveau total < fundingGoal:
//        - Changer project.status de FUNDED à ACTIVE
//        - Supprimer tout payout en attente (normalement inexistant, check défensif)
//        - Notifier le créateur que l’objectif n’est plus atteint
//        - Autoriser de nouveaux investissements
//    4. Si le nouveau total >= fundingGoal:
//        - Garder project.status = FUNDED (attente des autres périodes de grâce)
// ============================================================================

// ============================================================================
// TRANSITIONS D’ÉTAT (incluant la relance)
// ============================================================================
// Transitions autorisées:
//   INITIATED → SUCCESS, FAILED, CANCELLING
//   CANCELLING → CANCELLED, FAILED (stuck)
//   SUCCESS → REFUNDED, (annulation possible uniquement pendant la période de grâce)
//   FAILED → INITIATED   // NEW: relance d’un paiement échoué par l’utilisateur
//   (autres transitions interdites)
//
// Flux de relance (même Investment):
//   1. Investment.status = FAILED, paymentAttempts = N (e.g., 1)
//   2. L’utilisateur relance → status repasse à INITIATED
//   3. Le backend crée une nouvelle Transaction avec attemptNumber = N+1
//   4. paymentAttempts incremented to N+1
//   5. Le cron utilise le createdAt de la dernière transaction pour l’expiration
// ============================================================================

// ============================================================================
// RÈGLES DE VALIDATION D’ANNULATION
// ============================================================================
// Un utilisateur peut demander l’annulation d’un investissement si:
//
// 1. Investment.status = "INITIATED" (paiement pas encore complété).
//    → Annulation toujours autorisée (le provider peut encore l’accepter).
//
// 2. Investment.status = "SUCCESS" ET
//    Transaction.createdAt > (now() - cancellationGracePeriodMinutes * 60 * 1000)
//    → Annulation autorisée pendant la période de grâce (le provider peut accepter).
//
// 3. Investment.status = "SUCCESS" ET période de grâce expirée:
//    → Annulation REJETÉE. L’utilisateur doit demander un remboursement via support
//      (en suivant le flux de remboursement existant, incluant FailedRefundEvent).
//
// 4. Tout autre statut (CANCELLING, CANCELLED, FAILED, REFUNDED) :
//    → Annulation interdite.
// ============================================================================

// ============================================================================
// POLITIQUE DE MULTIPLES INVESTISSEMENTS
// ============================================================================
// NOTE: plusieurs investissements du même investisseur sur le même projet sont AUTORISÉS.
// Pas d’index unique (investorId, projectId) — intentionnel.
// Un investisseur peut contribuer plusieurs fois avec des montants distincts.
// ============================================================================

// ============================================================================
// RÉFÉRENCE POLITIQUE DE SUPPRESSION DE COMPTE
// ============================================================================

// ============================================================================
// DÉTECTION & RÉCUPÉRATION DES ÉTATS BLOQUÉS
// ============================================================================
//
// **Investissements CANCELLING bloqués :**
// Cron toutes les 15 minutes : GET /internal/cleanup-stuck-states
// Requête : { status: "CANCELLING", cancellingStartedAt: { $lt: Date.now() - 30*60*1000 } }
// Action :
//   1. Créer FailedCancellationEvent avec reason="STUCK_CANCELLING"
//   2. Repasser investment.status à "FAILED" (intervention admin manuelle requise)
//   3. Notifier l’admin
//
// **Flags scheduledForDeactivation « stale » :**
// Cron toutes les 15 minutes : GET /internal/cleanup-stuck-states
// Requête : { scheduledForDeactivation: true, updatedAt: { $lt: Date.now() - 5*60*1000 } }
// Action :
//   1. Logger un warning pour chaque flag stale
//   2. Si âge > 15 min : notifier l’admin (échec probable du workflow)
//   3. Remettre scheduledForDeactivation à false
//   4. Tracer l’événement dans AuditLog
//
// **Chemin de récupération:**
// L’admin examine FailedCancellationEvent et manuellement:
//   - Relance l’annulation via UC_RetryCancellation, OU
//   - Marque CANCELLED si le provider confirme, OU
//   - Marque SUCCESS si la fenêtre d’annulation est expirée, OU
//   - Marque FAILED (réconciliation provider manuelle requise)
// ============================================================================
