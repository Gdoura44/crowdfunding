// ============================================================================
// SCÉNARIOS D’ÉCHEC DE REMBOURSEMENT PAR RAISON
// ============================================================================
//
// **1. OVERFUNDING:**
// - Déclenché si: un investissement ferait dépasser currentFunding au-delà de fundingGoal.
// - Moment: traité synchronement dans le webhook de paiement AVANT toute incrémentation.
// - Montant remboursé: la **TOTALITÉ** de investment.amount (remboursement total), pas seulement l’excédent.
// - Comme le frontend empêche le surfinancement, c’est une vérification backend défensive.
// - Si le remboursement échoue :
//     • Investment.status = SUCCESS (fonds encaissés mais non remboursés)
//     • Transaction.refundStatus = FAILED, status = SUCCEEDED
//     • currentFunding NON incrémenté (le projet reste sous l’objectif)
//     • La clôture du projet est BLOQUÉE jusqu’au succès du remboursement ou intervention admin
//     • L’admin doit relancer le remboursement manuellement (UC_RetryRefund)
// - Récupération: l’admin relance le remboursement total. Si succès, investment devient REFUNDED.
//   Si la relance échoue de façon persistante, l’admin peut annuler côté provider
//   et marquer manuellement CANCELLED/REFUNDED dans le système.
//
// **2. EXPIRY :**
// - Déclenché si : la date limite du projet passe sans atteindre fundingGoal
// - Moment : traité par le cron expire-projects
// - Montant remboursé : investment.amount (total)
// - Si le remboursement échoue :
//     • Investment.status reste SUCCESS (pas REFUNDED)
//     • Transaction.refundStatus = FAILED
//     • L’admin doit relancer le remboursement manuellement (UC_RetryRefund)
//     • L’investisseur reçoit REFUND_FAILED.
//
// **3. USER_DEACTIVATED :**
// - Déclenché si : le compte investisseur est désactivé (isActive = false) pendant le traitement
// - Moment : traité synchronement dans le webhook
// - Montant remboursé : investment.amount (total)
// - Si le remboursement échoue : Investment SUCCESS, Transaction.refundStatus = FAILED,
//   relance manuelle requise.
//
// **4. ACCOUNT_DELETION :**
// - Déclenché si : l’investisseur supprime son compte (deletedAt renseigné)
// - Moment : traité pendant le flux de suppression de compte (bloquant)
// - Montant remboursé : investment.amount (total)
// - Si le remboursement échoue : la suppression est BLOQUÉE jusqu’à succès du remboursement
//   ou résolution admin (override possible après 7 jours, cf. user.txt).
//
// **5. PROJECT_DEACTIVATED :**
// - Déclenché si : l’admin désactive un projet ACTIVE (status → SUSPENDED) et que les investisseurs
//   ont droit à un remboursement.
// - Moment : traité synchronement dans le flux de désactivation admin.
// - Montant remboursé : investment.amount (total).
// - Si le remboursement échoue : Investment SUCCESS, Transaction.refundStatus = FAILED,
//   relance manuelle requise et REFUND_FAILED envoyée à l’investisseur.
//
// ============================================================================

// ============================================================================
// RÈGLE DE COHÉRENCE
// ============================================================================
// projectId DOIT être égal au projectId de l’Investment lié.
// Lors de la création d’un FailedRefundEvent, le système DOIT valider:
//   Investment.findById(investmentId).projectId == projectId
// En cas de mismatch, lever ValidationError.
// Cela évite la corruption de données et garantit un blocage correct de la clôture projet.
// ============================================================================

// ============================================================================
// MÉCANISME DE RELANCE (UC_RetryRefund)
// ============================================================================
//
// **Flux de relance (admin) :**
// 1. L’admin consulte l’événement FailedRefundEvent.
// 2. Il vérifie la raison pour comprendre le contexte.
// 3. Il clique « Relancer le remboursement ».
// 4. Backend :
//    - Charge la Transaction associée
//    - Vérifie Transaction.status = SUCCEEDED (la source du remboursement doit exister)
//    - Passe Transaction.refundStatus → "PENDING", incrémente refundAttempts
//    - Appelle l’API refund du provider
//    - En cas de succès :
//        • Transaction.refundStatus → SUCCEEDED
//        • Transaction.refundedAt = now
//        • Transaction.status → REFUNDED
//        • Investment.status → REFUNDED
//        • FailedRefundEvent.resolved = true ; resolvedBy = admin._id ; resolvedAt = now
//        • L’utilisateur reçoit PAYMENT_REFUNDED.
//    - En cas d’échec :
//        • retryCount++
//        • Transaction.refundAttempts++ (déjà incrémenté)
//        • Transaction.lastRefundAttemptAt = now
//        • Transaction.refundStatus → FAILED
//        • Champ error mis à jour
//        • Si retryCount > 5 : escalade vers responsable plateforme.
//
// **Cas spécial — relance OVERFUNDING :**
// - Si le remboursement de surfinancement échoue, la clôture du projet est BLOQUÉE.
// - La relance est REQUISE avant toute autre opération sur le projet.
// - Politique : résolution sous 24h (non imposée par le schéma).
//
// **Résolution manuelle :**
// - Si le provider confirme un remboursement déjà traité (problème d’idempotence) :
//     • L’admin met à jour manuellement Transaction.refundStatus → SUCCEEDED
//     • L’admin marque FailedRefundEvent.resolved = true
//     • L’admin documente la résolution dans le champ notes.
//
// ============================================================================

// ============================================================================
// COHÉRENCE STATUTS REFUND / INVESTMENT
// ============================================================================
//
// **Combinaisons valides :**
//  Investment.status=SUCCESS + Transaction.refundStatus=NOT_ATTEMPTED (normal)
//  Investment.status=SUCCESS + Transaction.refundStatus=PENDING (remboursement en cours)
//  Investment.status=REFUNDED + Transaction.refundStatus=SUCCEEDED (remboursement terminé)
//  Investment.status=SUCCESS + Transaction.refundStatus=FAILED (échec, relance nécessaire)
//
// **Combinaisons interdites (vérifiées côté service) :**
//  Investment.status=REFUNDED + Transaction.refundStatus=NOT_ATTEMPTED
//  Investment.status=REFUNDED + Transaction.refundStatus=PENDING
//  Investment.status=REFUNDED + Transaction.refundStatus=FAILED
//
// **Règle de transition :**
// Investment.status ne peut passer à REFUNDED que si Transaction.refundStatus = SUCCEEDED.
// ============================================================================
