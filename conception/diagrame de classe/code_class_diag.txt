note right of ProjectStatus
  **Transitions:**
  DRAFT → AWAITING_AI (creator submits project)
  AWAITING_AI → remains AWAITING_AI with aiStatus=FAILED (after 3 automatic n8n retries)
  AWAITING_AI → UNDER_REVIEW (after AI analysis completes)
  UNDER_REVIEW → APPROVED / REJECTED (admin decision)
  APPROVED → ACTIVE (admin publishes)
  ACTIVE → FUNDED (when currentFunding >= fundingGoal, goal reached but
    cancellation windows pending)
  FUNDED → CLOSED (when all SUCCESS investments are outside their cancellation grace period)
  FUNDED → ACTIVE (if a cancellation reduces effective funding below fundingGoal)
  ACTIVE → SUSPENDED (admin deactivates)
  REJECTED → DRAFT (creator edits and re-saves)
  DRAFT → AWAITING_AI (creator resubmits after rejection)
  SUSPENDED → UNDER_REVIEW (admin reactivates, AI still fresh)
  SUSPENDED → AWAITING_AI (admin reactivates, AI outdated: analyzedAt < (now - 30 days))

**Goal reached handling (no immediate payout):**
- When currentFunding >= fundingGoal, project.status becomes FUNDED.
- In FUNDED state, new investments are rejected (frontend disabled, backend 400).
- A cron job runs every minute to check FUNDED projects:
    • For each project, query all investments with status = SUCCESS.
    • For each such investment, get its Transaction.createdAt and the project's
      cancellationGracePeriodMinutes (stored per investment, not globally).
    • If ANY investment has Transaction.createdAt + cancellationGracePeriodMinutes > now(),
      the project remains FUNDED (still cancellable).
    • When ALL SUCCESS investments are aged beyond their grace period,
      transition project.status → CLOSED and create Payout (amount = fundingGoal).
- If a cancellation occurs while in FUNDED state:
    • Cancel the investment normally (status → CANCELLED, transaction → CANCELLED).
    • Recalculate effective total = sum of amounts of remaining SUCCESS investments.
    • If effective total < fundingGoal:
        - Set project.status = ACTIVE
        - Send notification to creator ("Funding goal no longer met")
        - Allow new investments
    • If effective total >= fundingGoal:
        - Remain in FUNDED (still waiting for other grace periods)
- The project's currentFunding field is NEVER decreased (historical record).
  -Effective funding for goal‑reached checks and cancellation eligibility is calculated
  -on the fly from SUCCESS investments whose linked Transaction has
   refundStatus IN ["NOT_ATTEMPTED", "SUCCEEDED"].
  -Investments with refundStatus = FAILED (overfunding or other reasons) are excluded.

  **Archiving rules:**
  Only DRAFT, AWAITING_AI, UNDER_REVIEW, or REJECTED projects can be archived.
  Archived projects are hidden from public search but visible on creator dashboard.
  ACTIVE, APPROVED, CLOSED, and SUSPENDED projects cannot be archived.
end note

note right of Project
  aiStatus:
  PENDING → no analysis yet (new or resubmitted project)
  COMPLETED → aiAnalysis embedded and valid
  FAILED → aiAnalysis null, manual retry required

    **AI Retry Logic:**
  - n8n workflow has its own retries (max 3, exponential backoff). After exhaustion,
    n8n calls /internal/mark-ai-failed → aiStatus = FAILED, creates FailedWorkflowEvent.
  - Admin can retry manually via dashboard. Each manual retry increments aiAnalysisRetries
    and resets aiStatus to PENDING.
  - Manual retries are limited to a maximum of 3 (aiAnalysisRetries < 3). After 3 failures,
    further retries are blocked; admin must manually review the project.
  - Project.status is NEVER automatically changed from AWAITING_AI to DRAFT based on retry count.
  The note in ProjectStatus transition is corrected.

  currentFunding is atomically incremented on each successful investment.
  currentFunding is capped at fundingGoal (enforced atomically via $inc with $max check).
  currentFunding is NEVER reduced after refunds — preserved for audit history.

    publishedAt is set when status transitions to ACTIVE.
  fundedAt is set when currentFunding reaches fundingGoal and status → FUNDED.
  (Because overfunding is fully refunded, currentFunding never exceeds fundingGoal – it stops exactly at the goal.)

  After CLOSED by goal: Payout is created automatically with
    amount = fundingGoal (always, regardless of overfunding).
  After CLOSED by deadline without reaching goal: all SUCCESS investments are refunded,
    no Payout is created.

  Overfunding: if a payment causes currentFunding to exceed fundingGoal,
    the funding is refunded to that investor (OVERFUND_REFUNDED notification),
    and the Payout is created only after overfunding refund completes successfully.
    
    Project closure (status → CLOSED) is blocked until all overfunding refunds have succeeded
    (i.e., no unresolved FailedRefundEvent with reason=OVERFUNDING for this project).

  **AI staleness rule:**
  AI analysis is considered outdated if analyzedAt < (now - 30 days).
  Used when admin reactivates a SUSPENDED project to decide transition path.

    Validation rule:
  When submitting a project (DRAFT → AWAITING_AI), deadline MUST be a future date.
end note

note right of Payout
  Created automatically when project status → CLOSED (goal reached).
  amount = fundingGoal at time of closure (never currentFunding, to exclude overfunding).
  creatorId is copied from Project.creatorId at creation time for direct audit reference.

  Status flow:
  PENDING → creator provides bank details → READY
  READY → admin approves → COMPLETED
  Any status → CANCELLED (admin cancels, or creator deleted before providing bank details)
  COMPLETED/READY → FAILED (bank transfer error, see FailedPayoutEvent)

  If isCreatorDeleted becomes true while Payout.status = PENDING or READY:
    Admin must manually cancel or complete the payout.
    Status → CANCELLED with a note explaining the reason.

  Bank details stored only for this payout, encrypted at rest (AES-256-CBC).
  Bank details are NOT stored in the User profile.
end note

note right of Investment
  Status flow:
  INITIATED → SUCCESS (payment confirmed by webhook)
  INITIATED → FAILED (payment failed or expired after 30 min by cron)
  INITIATED → CANCELLING (user requests cancellation while Transaction is PENDING)
  CANCELLING → CANCELLED (cancellation confirmed by provider)
  CANCELLING → CANCELLING stuck (cancellation failed → FailedCancellationEvent created)
  SUCCESS → REFUNDED (project expired without goal, or admin deactivates active project)

  Cancellation is allowed if:
    - Investment.status = INITIATED → always allowed.
    - Investment.status = SUCCESS AND Transaction.createdAt > (now() - cancellationGracePeriodMinutes * 60 * 1000) → allowed within grace period.
    - Otherwise (SUCCESS outside grace period, or any other status) → cancellation not allowed (user must request refund via support).

  Investment.status = INITIATED (payment not completed) → always allowed.

Investment.status = SUCCESS AND Transaction.createdAt > (now() - cancellationGracePeriodMinutes * 60 * 1000) → allowed within grace period.

Otherwise (SUCCESS outside grace period, or any other status) → cancellation not allowed (user must request refund via support).
  scheduledForDeactivation flag is set during async bulk project deactivation
    to prevent new state changes on an investment being processed.

  paymentAttempts counts total attempts for this Investment.
  Retry uses the SAME Investment document.
  A new Transaction is created with attemptNumber = paymentAttempts + 1,
  and paymentAttempts is incremented atomically.
  The cron uses the latest transaction's createdAt for expiry.

  cancellingStartedAt is set when status transitions to CANCELLING.
  Used by cleanup-stuck-states cron to detect investments stuck in CANCELLING > 30 min.

  Multiple investments by the same investor in the same project are allowed.

    **Stuck CANCELLING recovery:**
  The reconciliation cron (cleanup-stuck-states) also queries investments with
    status = CANCELLING AND cancellingStartedAt < (now - 30 min).
    These are flagged: a FailedCancellationEvent is created (reason = STUCK_CANCELLING)
    and a CANCELLATION_FAILED notification is sent to the investor.
  Admin resolves via the dead-letter dashboard.

note right of User
  Max 5 refresh tokens (device-aware).
  Oldest token removed when limit is exceeded.

  **Deactivation policy:**
  - isActive set to false.
  - INITIATED investments are cancelled automatically (payment provider cancellation attempted).
  - SUCCESS investments are NOT refunded.
  - Projects created by the user are NOT automatically deactivated.
  - Admin must manually deactivate projects if needed.
  - Bank details are NOT stored in User profile.

    **Deletion policy:**
  - Personal data is anonymized (email, name, phone replaced with tokens).
  - INITIATED investments are cancelled; SUCCESS investments are refunded.
  - For ALL projects: isCreatorDeleted = true, creatorId retained for audit.
  - Financial records (investments, transactions, payouts) retained with anonymized reference.
  - deletedAt timestamp set on the User document.
  - Deletion blocked if user has:
    • Pending payouts (PENDING/READY)
    • Active investments (INITIATED/CANCELLING)
    • Any SUCCESS investment whose linked transaction is not yet fully refunded (refundStatus != SUCCEEDED)
    • Any project they created with status in ["ACTIVE", "APPROVED", "UNDER_REVIEW", "AWAITING_AI", "FUNDED"]

end note

note right of Transaction
  Transaction is created AFTER the payment provider responds with a providerPaymentId.
  This guarantees providerPaymentId is always available at insert time.
  providerPaymentId is required and unique per (provider, providerPaymentId).

  refundStatus tracks the refund sub-lifecycle independently of status:
  NOT_ATTEMPTED → PENDING → SUCCEEDED / FAILED
  A transaction with status=SUCCEEDED can have refundStatus=FAILED without
    corrupting the original payment record.

  refundedAt MUST be null unless refundStatus = SUCCEEDED.

  attemptNumber starts at 1 (first transaction for this investment).
  Each retry creates a NEW Transaction document with attemptNumber incremented.
end note

note right of Report
  A creator cannot submit a report on their own project.
  Enforced at the service layer: reject if reporterId === project.creatorId.
  resolvedBy references the Admin User who resolved or dismissed the report.
  resolution field is required when status → RESOLVED or DISMISSED.
end note

note right of FailedWorkflowEvent
  Dead-letter collection for failed n8n workflows.
  projectId is OPTIONAL — populated only when workflow is project-scoped
    (e.g. risk-analysis). Null for non-project workflows (e.g. notification failures).
  Used for manual retry via admin dashboard.
  workflowType examples: "risk-analysis", "notification".

  When auto‑resolved, resolvedBy should be set to a system account (e.g., ObjectId("system"))
  and resolvedAt set to the current timestamp.
  resolvedBy/resolvedAt track which admin resolved the failure.
  **payload** field contains workflow‑specific input data.
  For "risk-analysis": { projectId, title, description, category, fundingGoal, deadline }
  For "notification": { userId, notificationId, emailTemplate, recipientEmail }
  Required for manual retry.
end note

note right of FailedCancellationEvent
  Dead-letter collection for failed payment cancellations.
  reason enum: CANCELLATION_FAILED (provider rejected), POST_UPDATE_FAILED (DB update failed), STUCK_CANCELLING (investment stuck in CANCELLING > 30 min).
  Also created automatically by the cleanup-stuck-states cron for stuck CANCELLING investments.
  Used for manual admin intervention.
end note

note right of FailedRefundEvent
  Dead-letter collection for failed refunds.
  reason enum: OVERFUNDING, EXPIRY, USER_DEACTIVATED, ACCOUNT_DELETION, PROJECT_DEACTIVATED.
  Used for manual admin intervention.
end note

note right of FailedPayoutEvent
  Dead-letter collection for failed bank transfers.
  Used for manual admin intervention.
end note
