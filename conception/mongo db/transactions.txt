// ============================================================================
// STATUS vs REFUND_STATUS STATE TRANSITION RULES
// ============================================================================
//
// **Core Principle:**
// - `status` tracks the PAYMENT lifecycle
// - `refundStatus` tracks the REFUND lifecycle (independent but related)
//
// **Transition Rules:**
//
// 1. **Payment Flow:**
//    PENDING → SUCCEEDED (webhook confirms payment)
//    PENDING → FAILED (webhook reports failure or timeout)
//    PENDING → CANCELLED (cancellation confirmed by provider;
//    the corresponding Investment.status may be CANCELLING while the cancellation is in progress,
//    but Transaction.status never enters CANCELLING)
//
// 2. **Refund Flow (starting from status=SUCCEEDED):**
//    refundStatus: NOT_ATTEMPTED → PENDING (refund request initiated)
//    refundStatus: PENDING → SUCCEEDED (provider confirms refund)
//    refundStatus: PENDING → FAILED (provider rejects refund)
//    refundStatus: FAILED → PENDING (admin manual retry)
//
// 3. **Authoritative State Rules:**
//    - status CAN be "REFUNDED" ONLY when refundStatus = "SUCCEEDED"
//    - When refundStatus → SUCCEEDED:
//        • status → "REFUNDED"
//        • refundedAt → current timestamp
//        • Investment.status → "REFUNDED"
//    - If refundStatus = "FAILED":
//        • status remains previous value (SUCCEEDED or FAILED)
//        • FailedRefundEvent created for admin review
//        • Investment.status remains unchanged until resolved
//
// 4. **Edge Case - Multiple Refund Attempts:**
//    - refundAttempts incremented each time refund is initiated
//    - lastRefundAttemptAt updated each attempt
//    - Admin can retry indefinitely via UC_RetryRefund
//    - Each retry: refundStatus FAILED → PENDING, refundAttempts++
//
// 5. **Invalid State Combinations (prevented at service layer):**
//     status="REFUNDED" + refundStatus="NOT_ATTEMPTED" (impossible)
//     status="REFUNDED" + refundStatus="PENDING" (refund must be confirmed)
//     status="REFUNDED" + refundStatus="FAILED" (contradictory)
//     status="SUCCEEDED" + refundStatus="SUCCEEDED" → transition to status="REFUNDED"
//
// 6. **Invalid refundStatus for failed payments:**
//    If transaction.status = "FAILED", refundStatus MUST remain "NOT_ATTEMPTED".
//    No refund shall be initiated for a failed payment. Service layer MUST enforce this.
//
// **Retry Logic:**
// - Payment retry: Create NEW Transaction under same Investment, increment attemptNumber.
// - Refund retry: Update SAME Transaction, increment refundAttempts.
//
// ============================================================================

// ============================================================================
// RETRY MODEL (clarification)
// ============================================================================
// - One Investment can have multiple Transactions (retries).
// - Investment.paymentAttempts tracks total attempts.
// - Each Transaction.attemptNumber corresponds to the attempt sequence.
// - When a payment succeeds, the Investment.status becomes SUCCESS,
//   and no further Transactions are created for that Investment.
// - Failed payments leave the Investment in FAILED state; user may retry,
//   creating a new Transaction under the same Investment.
// ============================================================================
