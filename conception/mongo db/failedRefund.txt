// ============================================================================
// REFUND FAILURE SCENARIOS BY REASON
// ============================================================================
//
// **1. OVERFUNDING:**
// - Triggered when: An investment would cause currentFunding to exceed fundingGoal.
// - Timing: Processed synchronously in the payment webhook BEFORE any increment.
// - Refund amount: The **ENTIRE** investment amount (full refund), not just the excess.
// - Because frontend prevents overfunding, this is a defensive backend check.
// - If refund fails:
//     • Investment.status = SUCCESS (money taken but not refunded)
//     • Transaction.refundStatus = FAILED, status = SUCCEEDED
//     • currentFunding is NOT incremented (project remains below goal)
//     • Project closure is BLOCKED until refund succeeds or admin intervenes
//     • Admin must retry refund manually (UC_RetryRefund)
// - Recovery: Admin retries full refund. If retry succeeds, investment becomes REFUNDED.
//   If retry consistently fails, admin may cancel the investment via provider dashboard
//   and manually mark it as CANCELLED/REFUNDED in the system.
//
// **2. EXPIRY:**
// - Triggered when: Project deadline passes without reaching fundingGoal
// - Timing: Processed by expire-projects cron job
// - Refund amount: Full investment.amount
// - If refund fails:
//     • Investment.status remains SUCCESS (not REFUNDED)
//     • Transaction.refundStatus = FAILED
//     • Admin must retry refund manually (UC_RetryRefund)
//     • User receives REFUND_FAILED notification
//
// **3. USER_DEACTIVATED:**
// - Triggered when: Investor account is deactivated (isActive = false) while payment is processed
// - Timing: Processed synchronously in webhook
// - Refund amount: Full investment.amount
// - If refund fails:
//     • Investment.status remains SUCCESS
//     • Transaction.refundStatus = FAILED
//     • Admin must retry refund manually
//
// **4. ACCOUNT_DELETION:**
// - Triggered when: Investor deletes account (deletedAt set)
// - Timing: Processed during account deletion flow (blocking)
// - Refund amount: Full investment.amount
// - If refund fails:
//     • Deletion is BLOCKED until refund succeeds or admin resolves
//     • Admin must retry refund manually
//
// **5. PROJECT_DEACTIVATED:**
// - Triggered when: Admin deactivates an ACTIVE project (status → SUSPENDED)
//   and investors are entitled to refunds.
// - Timing: Processed synchronously in the admin deactivation flow.
// - Refund amount: Full investment.amount.
// - If refund fails:
//     • Investment.status remains SUCCESS
//     • Transaction.refundStatus = FAILED
//     • Admin must retry refund manually (UC_RetryRefund)
//     • Investor receives REFUND_FAILED notification
//
// ============================================================================

// ============================================================================
// CONSISTENCY RULE
// ============================================================================
// projectId MUST equal the projectId of the linked Investment.
// When creating a FailedRefundEvent, the system MUST validate that:
//   Investment.findById(investmentId).projectId == projectId
// If mismatch, throw ValidationError.
// This prevents data corruption and ensures accurate project closure blocking.
// ============================================================================

// ============================================================================
// RETRY MECHANISM (UC_RetryRefund)
// ============================================================================
//
// **Admin Retry Flow:**
// 1. Admin views FailedRefundEvent in dashboard
// 2. Admin checks reason to understand context
// 3. Admin clicks "Retry Refund"
// 4. Backend:
//    - Fetches associated Transaction
//    - Verifies Transaction.status = SUCCEEDED (refund source must exist)
//    - Sets Transaction.refundStatus = "PENDING", increments refundAttempts
//    - Calls provider refund API
//    - On success:
//        • Transaction.refundStatus → SUCCEEDED
//        • Transaction.refundedAt = now
//        • Transaction.status → REFUNDED
//        • Investment.status → REFUNDED
//        • FailedRefundEvent.resolved = true
//        • resolvedBy = admin._id, resolvedAt = now
//        • User receives PAYMENT_REFUNDED notification
//    - On failure:
//        • retryCount++
//        • Transaction.refundAttempts++ (already incremented)
//        • Transaction.lastRefundAttemptAt = now
//        • Transaction.refundStatus → FAILED
//        • error field updated
//        • If retryCount > 5: Escalate to senior admin
//
// **Special Case - OVERFUNDING Retry:**
// - If overfunding refund fails initially, project closure is BLOCKED
// - Retry is REQUIRED before any further project operations
// - Admin must resolve within 24 hours (policy, not enforced by schema)
// - If unresolved after 24h: Automatic escalation to platform manager
//
// **Manual Resolution:**
// - If provider confirms refund already processed (idempotency issue):
//     • Admin manually updates Transaction.refundStatus → SUCCEEDED
//     • Admin marks FailedRefundEvent.resolved = true
//     • Admin documents resolution in notes field
//
// ============================================================================

// ============================================================================
// REFUND vs INVESTMENT STATUS CONSISTENCY
// ============================================================================
//
// **Valid State Combinations:**
//  Investment.status=SUCCESS + Transaction.refundStatus=NOT_ATTEMPTED (normal)
//  Investment.status=SUCCESS + Transaction.refundStatus=PENDING (refund in progress)
//  Investment.status=REFUNDED + Transaction.refundStatus=SUCCEEDED (refund complete)
//  Investment.status=SUCCESS + Transaction.refundStatus=FAILED (refund failed, needs retry)
//
// **Invalid State Combinations (prevented at service layer):**
//  Investment.status=REFUNDED + Transaction.refundStatus=NOT_ATTEMPTED
//  Investment.status=REFUNDED + Transaction.refundStatus=PENDING
//  Investment.status=REFUNDED + Transaction.refundStatus=FAILED
//
// **Transition Rule:**
// Investment.status can ONLY transition to REFUNDED when Transaction.refundStatus = SUCCEEDED
//
// ============================================================================
