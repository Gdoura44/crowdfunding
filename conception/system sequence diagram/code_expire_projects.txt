note right of API

  **Refund Policy:**

- When a refund job succeeds, it will update:
    • transaction.status = "REFUNDED"
    • transaction.refundStatus = "SUCCEEDED"
    • investment.status = "REFUNDED"

  - When a project expires without reaching its goal, all successful investments are refunded.
  - Refunds are processed asynchronously via BullMQ jobs to prevent cron timeouts.
  - The `refundStatus` field on the Transaction ensures idempotency: only transactions with `NOT_ATTEMPTED` are queued.
  - Transactions with `refundStatus = FAILED` are not retried automatically by the cron.
  - Admin must manually retry failed refunds via UC_RetryRefund.
  - BullMQ job retries are independent (max 3) and create FailedRefundEvent after exhaustion.
  - The project's `currentFunding` field is **not** updated after refunds to preserve historical data.
  - The project status is set to `CLOSED` to indicate it is no longer active.
  - Cron runs every minute to minimize the window between deadline and expiry.

  - **Overfunding Refund Block:** A project whose deadline has passed but still has unresolved overfunding refunds (FailedRefundEvent with reason=OVERFUNDING and resolved=false) will NOT be closed. The cron skips such projects and logs a warning. Once the overfunding refund is resolved (by admin retry or manual fix), the next cron run will close the project.

  - **Notifications:** In‑app notification for creator (expired) and for each investor (refunded) are created. Email jobs are queued after commit. On refund failure, admins receive REFUND_FAILED notification.

**Close FUNDED projects cron (every minute):**
- Find projects with status = "FUNDED".
- For each project, find all investments with status = "SUCCESS".
- Among those, filter to keep only investments where the associated Transaction
  has refundStatus IN ["NOT_ATTEMPTED", "SUCCEEDED"].
- For each such investment, fetch the associated transaction (latest attempt).
- **Grace period source:** `investment.cancellationGracePeriodMinutes` (per‑investment field).
- If for ANY investment: transaction.createdAt + investment.cancellationGracePeriodMinutes > now(),
    → skip this project (still cancellable).
- Else (all investments are outside grace period):
    - **Check for unresolved overfunding refunds**
    - Find FailedRefundEvent where projectId = project.id AND reason = "OVERFUNDING" AND resolved = false
    - If found:
        - Log warning and skip this project (remain FUNDED)
        - Continue to next project
    - Else:
        - Start transaction
        - Update project.status = "CLOSED"
        - Create Payout document (amount = project.fundingGoal, status = PENDING)
        - Insert AuditLog {
            actorId: ObjectId("system_cron"),
            actorRole: "ADMIN",
            action: "CLOSE_FUNDED_PROJECT",
            targetType: "Project",
            targetId: project.id,
            details: { fundingGoal: project.fundingGoal, payoutId: newPayout._id }
          }
        - Create PROJECT_FUNDED notification for creator
        - Commit transaction
        - Enqueue email job for creator
        **Overfunding block:** If any unresolved OVERFUNDING FailedRefundEvent exists,
          the project stays FUNDED until an admin resolves the refund. This prevents
          payout creation before overfunding is fully settled.
end note
